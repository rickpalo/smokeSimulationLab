# How to Release BatchSimLab

> Repository and source folder are `BatchSimLab` (renamed from `SmokeSimLab` at v0.9.3).  Lowercase runtime identifiers (`smoke_settings`, `SMOKE_*`, `.smokesettings`) still use the legacy prefix for .blend/keymap compatibility.

## Repository structure

```
BatchSimLab/                         ← repo root
├── scripts/
│   └── BatchSimLab/                 ← THE addon source (tracked — edit here)
│       ├── __init__.py              ← addon code
│       ├── smoke_worker.py          ← headless batch worker
│       ├── smoke_launcher.py        ← crash-safe launcher wrapper
│       └── blender_manifest.toml    ← Blender 4.2+ extension manifest
├── tests/                           ← pytest suite
├── tools/                           ← analysis helpers (analyze_estim.py, …)
├── docs/                            ← GitHub Pages extension-update feed (index.json + *.zip)
├── dist/                            ← built extension zips (git-ignored)
├── README.md
└── .gitignore
```

There is a **single tracked source tree** at `scripts/BatchSimLab/` — you edit
it in place and release straight from it. There is **no** separate
`BatchSimLab/` release-copy directory and **no** copy step (an earlier version
of this doc described one; it never existed).

---

## Releasing a new version

### 1. Update the version number

Open `scripts/BatchSimLab/__init__.py` and bump the version in `bl_info`:

```python
bl_info = {
    ...
    "version": (0, 2, 21),   # <-- change this
    ...
}
```

Use the format `(major, minor, patch)`:
- **patch** (0.2.x) — small bug fixes, test additions, documentation updates
- **minor** (0.x.0) — new features, backwards compatible
- **major** (x.0.0) — breaking changes

Also update `WORKER_VERSION` in `smoke_worker.py` and `LAUNCHER_VERSION` in
`smoke_launcher.py` if those files changed, and keep `_EXPECTED_WORKER_VERSION`
/ `_EXPECTED_LAUNCHER_VERSION` in `__init__.py` in sync with them. Keep
`blender_manifest.toml`'s `version` in sync with `bl_info` too — the
`test_extension_manifest.py::TestManifestVersionSync` test enforces this.

### 2. Run tests

```bash
python -m pytest tests/
```

All tests must pass before tagging a release. (No copy step — you edit and
release `scripts/BatchSimLab/` directly.)

### 3. Commit your changes

```bash
git add scripts/BatchSimLab/
git commit -m "v0.2.21: description of what changed"
```

### 4. Tag the commit and push

```bash
git tag v0.2.21
git push origin main --tags
```

The tag must start with `v` followed by numbers, e.g. `v0.2.21`.
Pushing the tag triggers the GitHub Actions release workflow automatically.

### 5. Check the release on GitHub

Go to: https://github.com/rickpalo/BatchSimLab/releases

GitHub Actions will create the release and attach `SmokeSimLab.zip`
(the installable Blender addon) within a minute or two.

---

## If you need to redo a release tag

```bash
git tag -d v0.2.21                    # delete local tag
git push origin :refs/tags/v0.2.21   # delete remote tag
# then re-tag and push as normal
```

## Building the extension (Blender 4.2+)

Since v0.7.6 the addon ships a `blender_manifest.toml`, so it can be packaged as
a Blender **extension** (not just a legacy add-on).  `bl_info` is retained as a
fallback for Blender < 4.2.

**Validate** (catches manifest/field errors before building):

```sh
blender --command extension validate scripts/BatchSimLab
# expect: "Success parsing TOML in ..."
```

**Build** the installable zip:

```sh
blender --command extension build --source-dir scripts/BatchSimLab --output-dir dist
# produces dist/batchsimlab-<version>.zip  (manifest + the 3 .py files; no __pycache__)
```

`dist/` and `*.zip` are git-ignored — the zip is a release artifact, not source.
Keep `blender_manifest.toml`'s `version` in sync with `bl_info` (the
`test_extension_manifest.py::TestManifestVersionSync` test enforces this).

**Install (end user):** Blender 4.2+ → Edit → Preferences → Get Extensions →
▾ (top-right) → Install from Disk… → pick `batchsimlab-<version>.zip`.  The
worker/launcher `.py` files are bundled inside the zip; the addon copies them to
the chosen Output folder at Export time, so the subprocess pipeline works
identically whether installed as a legacy add-on or as an extension.

**Notes / gotchas:**
- Each `[permissions]` string is capped at **64 characters** by Blender.
- `tagline` is capped at 64 chars and must not end with punctuation.
- Add-on `tags` must come from Blender's fixed add-on vocabulary.
- The repo + source folder are `BatchSimLab` (renamed from `SmokeSimLab` at
  v0.9.3); the lowercase runtime identifiers still use the legacy `smoke` prefix.
  The extension `id` is `batchsimlab` and must not drift (it keys the install dir).
