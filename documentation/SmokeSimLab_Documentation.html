Overview

SmokeSimLab automates the process of exploring smoke simulation parameters in Blender. Instead of manually changing settings, baking, and rendering one at a time, you define parameter ranges in a panel, click Export Batch, then Run Batch, and every combination runs automatically.

For each job the addon: bakes the Mantaflow fluid simulation, renders an MP4 animation from a PNG frame sequence, renders a final-quality PNG still, updates in-scene text objects with the current parameter values, and logs everything to a CSV file and a performance JSON file.

Why separate processes? Blender crashes when bpy.ops.fluid.bake_all() is called from inside another Python operator or timer. Launching separate Blender processes avoids this entirely — each process owns its own memory and event loop. Blender 5.1 also removed the built-in FFMPEG output format; SmokeSimLab works around this by rendering a PNG sequence first, then calling an external ffmpeg to assemble the MP4.

Installation

Download SmokeSimLab.zip from the Releases page.
In Blender open Edit → Preferences → Add-ons.
Click Install and select SmokeSimLab.zip.
Enable SmokeSimLab in the add-on list.
Press N in the 3D Viewport and open the SmokeLab tab.
Both files must be installed together. The zip contains __init__.py and smoke_worker.py. If you manually copy only one file the Export Batch button will show an error.

Quick Start

Set up a smoke simulation scene with a Mantaflow fluid domain object.
Open the SmokeLab tab in the N-panel.
Set Domain Object to your fluid domain.
Set Output to a folder for results.
Configure parameter defaults and any ranges to sweep.
Save your .blend file — the batch launcher needs its absolute path.
Click Export Batch to write the job files.
Click Run Batch (or double-click run_smoke_batch.bat in Windows Explorer) to start processing. Progress bars appear in the panel.
Tip: The Run Batch button launches the batch in the background and returns control to Blender immediately. You can continue working in Blender while the batch runs in separate processes.

Parameter Reference

Resolution

The number of voxels along the longest side of the fluid domain. Blender's default is 32. Higher values produce more detail but bake times increase roughly as the cube of the resolution — doubling the resolution multiplies bake time by ~8.

Gas Parameters

All three gas parameters live in a single collapsible "Gas Parameters" section.

Parameter Blender Attribute Default Range Description
Vorticity d.vorticity 1.0 0 – ∞ Adds turbulent swirling detail. 0 = smooth rising column, higher = chaotic swirling smoke.
Buoyancy Density d.alpha 1.0 –5 – 5 Buoyant force from smoke density. Positive = smoke rises, negative = sinks.
Buoyancy Heat d.beta 1.0 –5 – 5 Buoyant force from temperature. Works with flow object Initial Temperature setting.

Dissolve

When enabled, smoke fades out over time rather than accumulating indefinitely. The enable checkbox appears in the section header.

Option Description
Dissolve Speed Number of frames until smoke fully dissolves. Blender default is 5.
Slow Dissolve Use logarithmic fade instead of linear — smoke lingers longer then fades quickly.

Noise

Adds a high-resolution turbulent detail layer on top of the base simulation. The enable checkbox appears in the section header.

Parameter Blender Attribute Default Description
Scale d.noise_scale 2 Upres factor — how many times finer the noise grid is vs the base grid.
Strength d.noise_strength 1.0 How strongly the noise detail affects the visual result.
Position Scale d.noise_pos_scale 1.0 Spatial frequency of the noise pattern. Higher = finer, more detailed noise.

Input Modes

Every parameter supports three input modes, selectable with the Range / List toggles:

Mode How to activate Behaviour
Default only Neither Range nor List enabled The default value is used for every job. No iteration on this parameter.
Range Enable the Range toggle Values are generated from Begin to End in increments of Step.
List Enable the List toggle Specific values entered by the user via the + / − buttons.
Tip: Range and List are mutually exclusive — enabling one automatically disables the other.

Iteration Modes

Limited Combinations

  • Vary one parameter at a time
  • All others stay at their default value
  • Total jobs = sum of range lengths
  • Best for initial exploration

All Combinations

  • Full Cartesian product of all ranges
  • Total jobs = product of range lengths
  • Can produce very many jobs quickly
  • Best for fine-tuning after exploration

Example

Suppose you define: Vorticity range [0.5, 1.0, 1.5] and Noise Strength range [0.5, 1.0, 1.5]. All other parameters at default.

Mode Jobs Description
Limited 6 3 vorticity jobs + 3 noise strength jobs
All 9 3 × 3 = every combination

With five parameters each having a 5-step range, Limited = 25 jobs vs All = 3,125 jobs. Limited Combinations is strongly recommended for initial parameter exploration.

Render Engine Options

Mode Speed Quality Requirements
Cycles GPU (default) Medium Path-traced Works in --background mode. Uses OptiX → CUDA → CPU fallback.
EEVEE Fast Rasterised Requires a visible Blender window (windowed mode). Cannot run headless.
Animation output: SmokeSimLab always renders a PNG sequence first, then calls an external ffmpeg to assemble the MP4. This is compatible with Blender 5.1+, which removed the built-in FFMPEG output format. Frame PNGs are saved to Renders/<name>_frames/ and deleted after the MP4 is assembled.
Playblast samples: The animation render uses 32 Cycles samples by default for speed. Edit smoke_worker.py and change scene.cycles.samples = 32 to adjust quality vs speed. Re-export the batch after editing.

Running a Batch

After clicking Export Batch, the Run Batch button appears in the panel. Clicking it launches run_smoke_batch.bat in the background and begins polling for progress.

Three Progress Bars

While a batch is running the panel shows three stacked progress bars, each tracking a different level of detail:

Subtask
Rendering (195 of 500)
Frame-level progress within the current stage
Job
Stage 2 of 4 (~12 min remaining)
Time estimate for the current job
Overall
Job 3 of 12
Total jobs completed
Bar Label What it measures
Bar 1 — Subtask Stage name + frame count During bake: number of VDB cache files written so far.
During render: number of newly-written PNG frames since the job started. Only frames written after the job began are counted, so pre-existing frames from previous runs are excluded.
Bar 2 — Job Stage X of 4 + ETA Estimated time remaining for the current job. Each stage (Setup, Bake, Animation, Still) uses a three-tier estimate: actual elapsed time when a real-time rate can be computed, then a resolution- and dimension-based prediction using historical performance constants, then a hard-coded default.
Bar 3 — Overall Job N of M How many jobs in the batch have finished.

Stages

Each job passes through four stages. Bar 2 advances as stages complete:

Stage Description
1 — Setup Blender starts, loads the .blend file, sets simulation parameters.
2 — Bake Mantaflow fluid simulation bake. Bar 1 shows VDB file count.
3 — Animation PNG sequence render → ffmpeg MP4 assembly. Bar 1 shows frame count.
4 — Still Final high-quality still PNG render.

Crash Recovery

When Blender crashes mid-job, Windows normally shows a crash dialog (Blender has stopped working) and suspends the process until the user clicks a button — freezing the entire batch.

SmokeSimLab v0.1.27 includes smoke_launcher.py, a Python wrapper that is copied to your output folder at Export time. The batch file calls it instead of Blender directly. The launcher monitors for the Windows crash dialog process (WerFault.exe), and when detected:

  • Saves %TEMP%\blender.crash.txt to jobs/<name>_crash_<timestamp>.txt so every crash log is preserved.
  • Silently dismisses the dialog and terminates Blender.
  • Marks the job as failed so the batch continues to the next job.
  • Writes a jobs/<name>.crashed marker file for reference.

The failed job will be picked up by Retry Failed Jobs or by Automatically Retry Failed Jobs if that option is enabled.

Uses Blender's bundled Python. The batch file records the full path to Blender's own Python executable at Export time, so no system Python or PATH configuration is required. Re-export if you move or update your Blender installation.

Retry Failed Jobs

If any jobs failed, a Retry Failed Jobs button appears after the batch completes. It re-runs only the failed jobs, automatically enabling Use Placeholders and Use Existing Cache for the retry run so that already-completed frames and cache data are reused.

Resume and Retry Options

Three checkboxes below the parameter section control how the batch handles existing data from previous runs.

Use Placeholders

When checked, any job whose output PNG already exists in the Renders/ folder is skipped entirely — both bake and render. This lets you resume an interrupted batch without re-doing finished work.

Tip: Use this when you stopped a batch partway through. Export Batch again, check Use Placeholders, then Run Batch to pick up where you left off.

Use Existing Cache

When checked, the worker inspects the cache directory before baking:

  • Cache complete — bake is skipped entirely; render proceeds normally.
  • Cache partial — bake resumes from the last checkpoint (no full rebake).
  • No cache — full bake runs as usual.
Resumable checkbox required. For a partial bake to resume rather than restart from frame 1, the Resumable checkbox in Blender's Physics Properties → Fluid → Cache section must be enabled before the original bake begins. The worker sets this automatically when Use Existing Cache is active, but if the original bake was started without it, the cache lacks checkpoint data and Blender will restart from frame 1 regardless.

Automatically Retry Failed Jobs

When checked, the addon automatically fires a retry run two seconds after the initial batch completes if any jobs reported errors. The retry always uses Use Placeholders and Use Existing Cache so that successful jobs are skipped and partial caches are resumed.

One retry only. The auto-retry fires once per initial batch run. It does not re-retry a retry run. If jobs continue to fail after the auto-retry, use the Retry Failed Jobs button manually.

Display Results

When Display Results is enabled in the panel, the addon creates a SmokeOutput collection in your scene after the batch finishes. Each completed job becomes a plane object displaying the rendered still PNG as a material, arranged in a grid for easy visual comparison.

The collection is rebuilt each time a batch or retry completes. To clear it, delete the SmokeOutput collection from the Outliner.

Text Objects

SmokeSimLab can update FONT (text) objects in your scene with the current job's parameter values before each render, so the values appear in the rendered output. This is useful for comparison sheets.

Expand the Text Objects section in the panel and enter the exact name of each FONT object in your scene. Objects not found are silently skipped.

Field Default Name Example Value
Resolution Resolution_Text Res: 64
Noise Noise_Text Noise: U-2 | St-1.0 | Scale-1.0
Dissolve Dissolve_Text Dissolve: Time: 50 | Slow-No
Bake Time Time_Text Bake: 5 min 23 sec
Tip: Text objects are updated in the background Blender process. The changes are written to the scene in memory and appear in the rendered frames, but are not saved back to your original .blend file.

Examples:

Performance Analysis

After each job the worker appends a record to perf_log.json in the output folder. These records capture the timing data needed to calibrate the progress bar ETA estimates.

perf_log.json Fields

Field Description
timestamp ISO 8601 date-time when the record was written
job_name Job filename stem
resolution Fluid domain resolution
frame_end Number of frames in the animation
bake_skipped true if the bake was skipped (Use Existing Cache + complete cache)
bake_seconds Total bake wall-clock time in seconds (null if skipped)
bake_secs_per_res3_frame Normalised bake rate: seconds / (resolution³ × frames). Used to calibrate _BAKE_RATE_PER_RES3_FRAME.
render_engine CYCLES or EEVEE
render_width / render_height Output resolution in pixels
frames_rendered Number of frames actually rendered (may be less than frame_end with placeholders)
render_seconds Total render wall-clock time in seconds (null if skipped)
render_secs_per_pixel_frame Normalised render rate: seconds / (width × height × frames). Used to calibrate _RENDER_RATE_CYCLES/EEVEE_PER_PIXEL_FRAME.

analyze_perf.py

The tools/analyze_perf.py script reads perf_log.json and prints statistics to help you decide whether to update the estimation constants in the addon.

Usage

python tools/analyze_perf.py <path_to_perf_log.json>

Output

For each metric the script prints: sample count, mean, standard deviation, median, coefficient of variation (CV), and a suggested constant to paste into __init__.py. A per-job actual vs. predicted bake table is also shown.

================================================================
SmokeSimLab Performance Analysis
File    : C:\...\perf_log.json
Records : 12
================================================================

BAKE  —  bake_time ≈ K × resolution³ × frames
  bake_secs_per_res3_frame
    Samples  : 9
    Mean     : 9.5400e-07
    Std Dev  : 8.2000e-08
    Median   : 9.4000e-07
    CV       : 8.6%  (good fit)
    → Paste into __init__.py:  _BAKE_RATE_PER_RES3_FRAME = 9.5400e-07
...
Tip: A CV below 20% means the linear model fits well and the mean is a reliable constant. A CV above 20% usually indicates mixed hardware, different scene complexity, or a temperature-throttled CPU. Collect more data at consistent settings before updating the constants.

Updating the Estimation Constants

Once you have at least 5–10 jobs of data, run analyze_perf.py and look at the suggested constants at the bottom of each section. If the CV is below 20%, paste the suggested values into scripts/SmokeSimLab/__init__.py: 

_BAKE_RATE_PER_RES3_FRAME           = 9.54e-7   # s / (res³ × frame)
_RENDER_RATE_CYCLES_PER_PIXEL_FRAME = 7.23e-9   # s / (pixel × frame)
_RENDER_RATE_EEVEE_PER_PIXEL_FRAME  = 3.61e-9   # s / (pixel × frame)

Re-install the addon from the updated zip after editing these constants.

Troubleshooting

Blender crash dialog freezes the batch

When Blender crashes, Windows suspends its process while showing a crash dialog, which blocks the batch from continuing. SmokeSimLab v0.1.27+ includes smoke_launcher.py which detects and dismisses this dialog automatically. Ensure Python is on your system PATH so the launcher can run. If the batch was exported with an older version, re-export to get the updated batch file that uses the launcher.

Crash logs accumulate in the jobs/ folder

Each detected Blender crash produces a job_NNNN_crash_YYYYMMDD_HHMMSS.txt file in the jobs/ directory copied from %TEMP%\blender.crash.txt. These files preserve the full stack trace for diagnosis. Review them to identify patterns (e.g., consistent crash in the same simulation stage or at the same resolution). They are safe to delete once reviewed.

Export Batch shows "smoke_worker.py not found"

Both files must be installed together. Re-install the addon using the zip file. Do not install __init__.py and smoke_worker.py separately.

Batch crashes immediately with EXCEPTION_ACCESS_VIOLATION

Some installed Blender addons crash when Blender loads in background mode. The --factory-startup flag prevents most user addons from loading. If this doesn't help, check jobs/job_0000.log — the last line before the crash usually identifies the problematic addon.

Use Existing Cache clears the bake and starts from frame 1

The bake can only resume from a checkpoint if the Resumable checkbox was enabled in Physics Properties → Fluid → Cache before the original bake began. Without it, Blender has no checkpoint data. To fix: enable Resumable, clear the cache, and re-bake from the start. From that point forward, partial bakes can be resumed.

Progress bar shows "Rendering (500 of 500)" while rendering is still in progress

This can occur if the output frames directory contains PNG files from a previous run. SmokeSimLab only counts frames written after the current job started (using file modification times). If you see this, verify that the frames directory does not contain stale PNGs from an earlier batch, or enable Use Placeholders so existing frames are skipped rather than re-rendered.

Bar 2 ETA is stuck and not counting down

Bar 2 switches to a real-time rate estimate once frames are actively rendering (frame count is between 0 and the target). If the frame count jumps immediately to 100% (stale PNGs scenario above), the rate estimate falls back to a countdown against the default estimate. The ETA becomes accurate once at least one frame has been genuinely rendered since job start.

KeyError: key "Fluid" not found in the console

A Text datablock script in your .blend file is trying to access obj.modifiers["Fluid"] by name, but the modifier has a different name. The SmokeSimLab worker is unaffected (it finds the fluid modifier by type, not name), but the error appears in the Blender console. To silence it: open the Text Editor in Blender, select the auto-run script, and uncheck Register in the header bar.

Cache files found: 0 after bake

The bake completed but wrote nothing to disk. Common causes:

  • Cache directory path contains special characters or spaces in unusual positions
  • The domain object's Fluid modifier is set to a type other than Domain
  • Output path is on a network drive that is slow to flush

Check the full log for error messages before the "Cache files found: 0" line.

Animation MP4 is missing or not created

SmokeSimLab requires ffmpeg.exe to be on the system PATH to assemble the MP4 from the PNG frame sequence. Install ffmpeg and ensure it is accessible from a command prompt (ffmpeg -version should print its version). The frame PNGs remain in the _frames/ subdirectory if the ffmpeg step fails.

Playblast takes a very long time

The animation render uses 32 Cycles samples by default. For faster previews, edit smoke_worker.py in your output folder and reduce scene.cycles.samples = 32 to 8 or 16. Re-export the batch after changing (Export Batch overwrites the worker copy).

Physics panel resolution is greyed out after batch

The domain is pointing at a job cache directory which has existing files, so Blender locks the parameters. In Physics Properties → Fluid, click Free All, then point the cache directory back to your working folder.

The job count seems wrong

In Limited Combinations mode the count is the sum of range lengths (not the product). In All Combinations mode it is the product. The panel shows the exact count above the Export button. Also check for the floating-point range off-by-one: a range of 0.2 to 1.0 in steps of 0.2 correctly produces 5 values (the epsilon fix handles this).

EEVEE renders crash or produce blank frames

EEVEE requires an OpenGL context which is not available in --background mode. Switch to Cycles GPU in the Render Engine selector, or if you specifically need EEVEE, switch to EEVEE mode which opens a small visible Blender window per job.
